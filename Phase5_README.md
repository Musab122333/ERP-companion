# Phase 5 — Web Dashboard (Next.js)

## What This Builds

A 5-page dashboard calling your Phase 4 API:

| Page | Route | Shows |
|---|---|---|
| Overview | `/` | KPI cards + monthly sales trend chart |
| Customers | `/customers` | Top customers table + outstanding receivables |
| Products | `/products` | Top products bar chart + slow-moving table |
| Finance | `/finance` | Cash flow chart + recent transactions |
| Inventory | `/inventory` | Full item master table |
| Invoice detail | `/invoice/[invoiceNo]` | Single invoice drill-down |

Every page respects a `source_folder` filter (ZF2526 / ZF2627 / ZFN2526 / All)
via the dropdown in the top-right, which persists across page navigation.

---

## Step 1 — Create the Next.js project (run this yourself)

```bash
npx create-next-app@latest dashboard --typescript --tailwind --eslint --app --src-dir --import-alias "@/*"
cd dashboard
```

## Step 2 — Install the extra packages

```bash
npm install recharts lucide-react
```

## Step 3 — Set up shadcn/ui

```bash
npx shadcn@latest init
```

Defaults are fine for every prompt.

Then install **all 8 components** used across these pages
(note: this is more than originally planned — `badge` and `table`
were needed in addition to the original 6):

```bash
npx shadcn@latest add card table badge button select tabs
```

## Step 4 — Copy these files into your project

## Step 5 — Configure the API URL

Copy `.env.local.example` to `.env.local` in your project root:

```
NEXT_PUBLIC_API_URL=http://localhost:8000
```

This should match wherever your Phase 4 FastAPI backend is running.

## Step 6 — Run it

Make sure your Phase 4 API is running first (in its own terminal):
```bash
uvicorn main:app --reload --host 0.0.0.0 --port 8000
```

Then in a **second terminal**, inside the `dashboard` folder:
```bash
npm run dev
```

Open **http://localhost:3000** in your browser.

## What you should see

**Overview page (`/`)** — 4 KPI cards (Total Revenue, Today's Sales, Active
Customers, Outstanding Receivables) followed by a line chart of monthly
sales trend. Use the company dropdown top-right to filter by ZF2526 /
ZF2627 / ZFN2526, or leave on "All Companies" to see combined totals.

**Customers page** — two side-by-side tables: top customers ranked by
purchase value, and outstanding receivables with an "Over Limit" badge
for anyone exceeding their credit limit.

**Products page** — a horizontal bar chart of your top 10 products by
revenue, followed by a table of slow-moving stock (no sales in 90+ days).

**Finance page** — a debit/credit bar chart by date, followed by a
recent transactions table with colored badges (red=Debit, green=Credit).

**Inventory page** — full item master, 100 rows, with Goods/Service badges.

**Invoice detail** — navigate by typing a URL directly, e.g.
`http://localhost:3000/invoice/263?source_folder=ZF2526` — shows that
exact invoice we used throughout Phase 2/3 debugging, full circle.

## Troubleshooting

| Symptom | Likely cause |
|---|---|
| "Couldn't load this data" on every page | FastAPI backend isn't running, or wrong `NEXT_PUBLIC_API_URL` |
| Blank/empty tables | Backend running but no rows match — check `source_folder` filter |
| Module not found: '@/components/ui/...' | Forgot to run `npx shadcn add` for that component |
| Module not found: '@/lib/utils' | `shadcn init` wasn't run, or file got overwritten |
| Charts don't render | `npm install recharts` step skipped |
| TypeScript errors about `searchParams` | Make sure you used `--app` flag — this code uses Next.js 15's App Router |

## Design notes

**Server Components by default.** Every page fetches data directly on
the server (no client-side loading spinners for initial page load) —
faster, and the API calls never touch the browser's network tab,
which also means no CORS headaches even though we left CORS open on
the FastAPI side.

**Suspense boundaries per-section.** Each table/chart loads
independently — if the customers table is slow but receivables loads
fast, you see the receivables data immediately while customers shows
a skeleton loader.

**Numbers come back as strings from the API.** PostgreSQL's NUMERIC
type gets serialized as a string by FastAPI/psycopg2 to avoid floating
point precision loss — the `formatCurrency`/`formatNumber` helpers
handle this conversion consistently everywhere, so you never need to
worry about it in page code.
